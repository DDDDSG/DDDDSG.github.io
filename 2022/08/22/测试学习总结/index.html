第一章 基本概念

测试主流技能

​ 功能测试：比如登录功能手动测试，保证功能正确实现

​ 接口测试：代码和工具对服务端提供的接口进行测试

​ 性能测试：代码和工具，模拟多人使用，查找服务器缺陷

​ 自动化测试：代码和工具代替手工，对项目进行测试，生成报告

常见的测试分类

​ 黑盒测试：源代码不可见，只能见到 UI 功能可见

​ 灰盒测试：部分源代码可见，UI 功能可见

​ 白盒测试：全部源代码可见，只能见到 UI 功能可见

软件测试流程

​ 1、需求分析；2、测试计划；3、用例设计；4、用例执行；5、缺陷管理；6、测试报告

测试用例

测试用例：为测试项目而设计的执行文档

​ 作用：防止漏测；实施测试的标准

用例设计编写格式 8 大要素：用例编号、用例标题、项目\模块、优先级、前置条件、测试步骤、测试数据、预期结果

​ 用例编号：项目_模块_编号；

​ 用例标题：预期结果（测试点）；

​ 模块/项目：所属模块或项目；

​ 优先级：P0~P4，P0重要程度最高

​ 前置条件：执行用例的前置操作

​ 测试步骤：描述操作步骤

​ 测试数据：被操作的数据

​ 预期结果：期望达到的结果

​ 实际结果：执行用例填写实际结果

第二章 测试设计

测试六大流程：需求分析、测试计划、用例编写、用例执行、缺陷管理、测试总结。

如何设计测试点

等价类划分

等价类划分：具有某种共同特征的数据集进行划分，可以划分为有效等价类和无效等价类。

​ 解决：能对穷举场景设计测试点

​ 步骤流程：1、明确需求；2、确定有效无效边界；3、提取数据编写测试用例

验证 QQ 账号的合法性（6~10位自然数） 案例：

qq_002 qq不合法（5位自然数） qq p1 打开qq程序 1、输入qq号；2、点击验证按钮 12345 不合法

qq_004 qq不合法（6位非自然数） qq p1 打开qq程序 1、输入qq号；2、点击验证按钮 123456a 不合法

边界值分析法

边界值分析法：能对限定边界规则设计测试点（>=，>，<，<=）

​ 上点：边界上的点

​ 内点：范围内的点

​ 离点：离边界最近的点

​ 步骤流程：1、明确需求；2、划分等级啊；3、确定边界；4、编写用例

判定表法

判定表法：能对多条件依赖关系进行设计测试点

​ 等价类划分，主要关注单个输入类条件测试。判定表法，考虑多个条件之间的组合。

​ 案例：

​ 步骤流程：1、明确需求；2、画判定表；3、编写用例

场景法：又称流程图法，能对项目业务进行设计测试点

第三章 缺陷管理

了解 it 行业常见岗位

​ 前端/后端/测试/ui/运维/产品/运营

缺陷判定标准

​ 少功能、多功能、功能错误、隐形功能错误、不易使用

​ BUG 类型：功能错误、界面错误、兼容性、数据、易用性、架构等

缺陷管理流程

​ 测试流程：需求分析、测试计划、用例设计、用例执行、缺陷管理、测试报告

​ BUG 的管理流程：提交缺陷、分派缺陷、处理缺陷、回归测试、关闭缺陷

​ 缺陷报告：缺陷 ID、缺陷标题、缺陷状态、严重程度、优先级、所属模块、缺陷描述、附件

使用工具对 BUG 进行管理

​ 状态：新建、打开、关闭、拒绝、延期

​ 缺陷描述：前置、步骤、预期、实际

Web 登录功能测试

登录需求分析-1

1、输入正确的中国手机号(11位)
当文本框失去焦点的时候验证，红色为失败，绿色为成功
2、点击发送验证码
如果手机号文本框状态为绿色，弹出”点击按钮进行验证”；
如果手机号文本框为红色，提示手机号不正确
3、点击按钮进行验证
拖拽图形到指定位置，按钮消失；
拖拽图形未到指定位置，晃动提醒，滑块回到初始位置； 超过5次，提示尝试过多，请点击重试；

登录需求分析-2

1、输入验证码
正确的验证码，并“勾选我已阅读并同意”，点击登录，进入系统;
错误的验证码，并“勾选我已阅读并同意”，点击登录，提示验证码错误:
正确的验证码，未“勾选我已阅读并同意”，点击登录，提示请勾选;

2、点击登录
手机号、验证码都为绿色，勾选“我已阅读并同意”，登录成功

登录测试点提取

测试点分析：正向尽量覆盖多条；逆向一次覆盖一条，其他必须正确

登录模块：手机号、点击验证按钮、输入验证码、勾选协议

​ 手机号：正向–11位正确手机号；逆向–为空\10位纯数字\12位纯数字\11位非数字\未注册（格式正确）

​ 点击验证按钮：正向–发送成功；逆向–发送失败

​ 输入验证码：正向–正确；逆向–为空\错误\过期

​ 勾选协议：正向–勾选；逆向–未勾选

​ UI 原型：1、布局、颜色与原型一致；2、图片文字准确无误

​ 兼容性：火狐、谷歌、IE/Edge、苹果

验证码获取功能：正向–发送成功；逆向–发送失败（未到位置）/发送失败（重试）

发布文章需求分析

1.文章标题不能小于5个字符
2.文章内容不能为空
3.频道不能为空 4.封面选择，单图、三图，无图、自动
5.点击选择图片
素材库、上传图片切换素材库
全部和收藏切换图片可以选择上传图片
点击选择图片-选择本地文件
点击开始上传-如果已经选择本地文件，点击上传，上传成功1点击开始上传-如果未选择本地文件，提示”请选择一张图片”

​ 内容–正向：内容为1个字符；逆向：内容为空

​ 封面–正向： 单图/三图/无图/自动；逆向：单图未选图/三图未选图

​ 频道–正向：选一个；逆向：为空

​ 结果–发表；存入草稿箱

​ 独立–1、选择图片：；2、选择频道

​ 测试六大流程：需求分析、测试计划、设计用例（1、设计测试点；2、编写用例覆盖测试点）、执行用例（填写实际结果）、缺陷管理（提交->开发修复->验证->关闭）、测试报告。

测试报告八大要素

​ 项目背景、测试目标、提测标准、结束标准、风险控制、bug统计、bug分析、测试总结

​ 项目背景：项目介绍

​ 测试目标：登录模块/发布文章模块

​ 提测标准：冒烟测试用例100%通过；被测试内容符合约定版本及功能

​ 结束标准：P0~P2 全部修复完成；P3 修复95%

​ 风险控制：人员风险，测试环境风险，需要风险

​ bug 统计：登录模块8个；发布文章1个（P0）

​ bug分析：略

​ 测试总结：问题：登录需求不明确；选择频道不明；上传图片功能有点问题；收获：先设计主功能；其次设计独立功能点；在设计测试用例前先设计测试点

第五章 性能测试

性能测试：基准测试、负载测试、稳定性测试、并发测试、压力测试

性能测试指标：响应时间、并发数、吞吐量（单位时间处理客户端请求数量）、点击数（不是点击一次的数量）、错误率（负载下，失败的概率）、资源利用率（CPU、内存、磁盘 IO、网络等资源的利用情况）

性能测试的流程：性能测试需求分析、性能测试计划和方案、性能用例设计、性能测试执行、性能分析和调优、性能报告总结

性能测试用例：用例名称、用例编号、用例描述、前置条件、用例步骤和期望结果、实际结果（并发数、响应时间、吞吐量、成功率各个指标与事务、服务器数性能、据库性能）

性能测试执行：建立测试环境、编写测试脚本、性能测试监控、执行测试脚本

JMeter 压力测试工具

使用工具模拟多个客户端访问服务器，查看性能。

1、创建线程组，用来模拟多个用户

2、创建 HTTP 请求，用于访问某功能

3、添加定时器，用于添加时间间隔否则服务器容易宕机

4、聚合报告，主要看吞吐量（处理多少请求/s）

优化前 10 左右的吞吐量，优化后（对网站性能优化采用：本地缓存+分布式缓存方案）达到了 190 左右的吞吐量。

单元测试

每次的测试步骤：初始化数据、执行测试代码、验证测试结果、清理测试数据
